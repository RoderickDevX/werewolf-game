# 手机端联机 MVP 设计规格

## 目标

先把联机玩法的第一段体验重建成手机优先的大厅和等待房间。当前阶段的目标不是一次性重做完整游戏界面，而是先让玩家在手机上能顺畅完成创建房间、加入房间、等待开局这条关键路径。

本轮改造后的基础流程是：

1. 打开游戏。
2. 选择昵称和头像。
3. 创建房间或加入已有房间。
4. 在清晰的 9 人等待房里等其他玩家。
5. 房主准备好后，沿用现有游戏逻辑开始对局。

## 当前问题

项目现在把三个方向混在了同一个页面里：

- 电脑端桌面牌桌布局。
- 联机房间大厅。
- 只完成了一部分的手机端适配。

这导致手机上很难看清当前应该做什么。大厅、等待房、开场海报、游戏牌桌、弹窗、轮询状态和行动渲染都挤在同一套 HTML、CSS 和 JavaScript 表面里。第一步修复应该先收窄范围，只处理玩家开局前必须使用的手机端入口流程。

## 范围

本设计只覆盖手机端大厅和等待房间。

本轮包含：

- 手机优先的大厅布局。
- 手机优先的等待房布局。
- 可加入房间卡片。
- 昵称和头像选择。
- 房间码展示和复制按钮。
- 9 个座位的等待房网格。
- 准备、开始、返回大厅等操作。
- 保持桌面浏览器也可用的响应式样式。
- 为了让大厅和等待房代码更容易维护而做的最小 JavaScript 整理。
- 保护房间流程和关键 DOM 结构的测试。

本轮不包含：

- 重做完整游戏回合界面的手机端体验。
- 引入 WebSocket。
- 引入数据库、账号或持久化登录。
- 修改狼人杀角色、胜负规则、阶段顺序或 AI 行为。
- 替换现有头像和图片资源。
- 把前端重写成 React、Vue 或其他框架。

## 产品方向

大厅要更像手机游戏入口，而不是桌面网页表单。

大厅应该具备：

- 小型顶部栏，显示游戏名和刷新操作。
- 靠上的玩家设置区。
- 方便点击的昵称输入框。
- 以可见头像按钮或头像卡片为主的头像选择，而不是只依赖桌面下拉框。
- 明确突出的创建房间按钮。
- 易扫读的可加入房间卡片。

等待房要更像开局前的组队房间。

等待房应该具备：

- 房间码是最明显、最方便分享的信息。
- 真人人数、AI 补位数、房主状态用紧凑标签展示。
- 座位以稳定的 3 x 3 网格展示。
- 真人座位展示头像、昵称、房主/准备状态，以及必要时的“我”标记。
- 空座位明确说明开局时会由 AI 补位。
- 底部操作栏在手机上始终容易触达。

## 页面设计

### 大厅

大厅包含：

- 顶部栏：`卡通狼人杀`、刷新按钮，以及轻量的在线或房间状态。
- 玩家设置区：昵称输入和头像选择。
- 主操作：创建房间。
- 可加入房间列表：每张卡片展示房间码、房主或玩家摘要、当前人数、AI 补位数和加入按钮。
- 空状态：没有可加入房间时，展示简短提示和创建房间入口。
- 加载和错误状态：用一个状态区反馈房间列表刷新、创建房间等操作结果。

房间列表在手机上不允许横向滚动，房间卡片应纵向排列。

### 等待房

等待房包含：

- 头部：返回按钮、房间码、复制按钮。
- 状态标签：真人数量、AI 补位数量、房主或玩家状态。
- 座位区：手机端固定为 3 列，共 9 个座位。
- 底部操作栏：
  - 返回大厅。
  - 准备或取消准备。
  - 开始游戏，仅房主在后端允许时可用。
- 状态消息：放在底部操作区上方或操作区内部。

底部操作栏不能遮挡座位内容。等待房页面需要为底部操作栏预留足够的底部间距。

## 状态和数据流

本轮继续使用现有 HTTP 房间接口和 localStorage 会话逻辑。

前端状态保持轻量：

- 当前房间 ID。
- 当前玩家 ID 或会话身份。
- 当前昵称和头像选择。
- 通过轮询拿到的最新房间快照。
- 当前操作中的 UI 状态，例如创建中、加入中、准备中、开始中或刷新中。

大厅负责获取可加入房间。创建或加入成功后进入等待房。等待房通过轮询更新房间快照，并根据最新房间数据重新渲染座位网格。

不要在 DOM 里引入第二套真实状态。大厅卡片和等待房座位都应该从最新房间数据渲染出来。

## 代码边界

实现要保持增量，不做大重写，但需要把职责分清楚：

- 大厅渲染和大厅事件处理集中放在一起。
- 等待房渲染和等待房事件处理集中放在一起。
- 共享辅助函数负责头像查找、状态消息、房间会话存取等小功能。
- 游戏回合内的渲染逻辑本轮尽量不动。

如果现有 `app.js` 已经难以安全编辑，先在同一个文件内把大厅和等待房逻辑整理成小函数。是否拆成多个前端文件留到后续阶段处理。

## 错误处理

面向玩家的错误信息要短、明确、可行动：

- 房间加载失败：提示玩家重试。
- 房间已满或已开局：返回大厅并刷新房间列表。
- 昵称、头像或输入不合法：停留在当前页面，并在状态区提示。
- 非房主开始游戏被拒绝：停留在等待房，并提示只有房主可以开始。
- 轮询时网络失败：保留上一次房间快照，同时显示正在重连或刷新失败状态。

前端绝不能暴露原始异常堆栈或 API Key。

## 响应式要求

主要目标设备：手机竖屏，约 390 x 844。

次要目标：

- 小屏手机，约 360 px 宽。
- 手机横屏或小平板，约 800 px 宽。
- 桌面浏览器仍然可用，可以居中显示手机流程，或在空间足够时使用适度的双栏大厅。

本轮结束后，游戏回合界面在手机上可以仍然不完美，但大厅和等待房必须是可用、清晰、完整的。

## 视觉规则

- 使用现有卡通头像资源。
- 控件要适合触屏点击，高度稳定。
- 大厅和等待房里不要再使用开屏宣传页式的大标题。
- 不要出现卡片套卡片。
- 卡片圆角保持克制。
- 不添加装饰性渐变光斑或纯装饰背景。
- 手机端文字对比度要清楚。
- 本轮不为了图标引入新的依赖；如果没有现成图标库，优先使用文字按钮或简单符号。

## 测试和验证

自动化测试需要覆盖：

- 大厅页面包含房间列表、创建房间、昵称、头像选择等关键控件。
- 等待房包含房间码、复制按钮、9 个座位、准备按钮、开始按钮和返回按钮。
- 创建房间和加入房间仍然遵守现有后端规则。
- 已开局或已满的房间不会出现在可加入列表。
- 只有房主可以开始游戏。
- 现有测试套件仍然通过。

手动浏览器验证需要覆盖：

- 390 x 844 手机视口。
- 360 x 740 小屏手机视口。
- 1280 x 720 桌面视口。
- 没有可加入房间的大厅。
- 有一个可加入房间的大厅。
- 房主视角的等待房。
- 如果本地会话模拟可行，再验证非房主视角的等待房。

## 成功标准

本轮成功的标准是：

- 手机用户打开第一屏后能立刻理解下一步该做什么。
- 创建房间或加入房间有一条清晰主路径。
- 等待房能清楚展示全部 9 个座位和 AI 补位规则。
- 等待房主要操作在手机底部容易点击。
- 仍然可以使用现有后端和现有游戏回合逻辑开始游戏。
- 现有测试通过，并新增测试保护手机端大厅和等待房结构。
